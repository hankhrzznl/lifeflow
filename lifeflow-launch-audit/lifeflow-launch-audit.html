总体评估：LifeFlow 核心功能（首页待办、日程管理、记账、目标系统、AI 助手）已具备可上线质量，但存在 4 个 P0 阻断问题——路由死链和缺少防重复提交防护——可能直接导致用户数据异常和页面白屏。建议在修复所有 P0 项后上线，P1 项可在上线首周跟进。产品信息架构方面，当前存在功能孤岛和导航断层，Part 2 提供了一份完整的重构方案，建议分阶段实施。

Part 1: 健壮性审查报告

P0 阻断问题（4项）

P0 #1 growth 和 learning 路由导向 404

位置：src/app/growth/page.tsx 和 src/app/learning/page.tsx

原因：两个页面均使用 router.replace("/overview") 做客户端跳转，但 /overview 路由在 Next.js App Router 中并不存在，用户会被重定向到一张空白 404 页面。

风险：如果底部导航或任意内部链接指向这两个路径，用户将遭遇白屏无提示的 404。

修复建议：将跳转目标改为 /（首页），或完全删除这两个路由并在引用处替换链接；或使用 Next.js redirect() 在服务端完成 301 跳转。

P0 #2 首页新建事项无防重复提交

位置：src/app/page.tsx，handleCreate 函数（第 167-184 行）

风险：当前逻辑仅校验了标题和时间非空，未设 isSubmitting 锁。用户快速连续点击"添加事项"按钮将触发多次 addManualItem 调用，导致数据库中出现完全相同的重复事项记录。这在性能较慢的设备或网络环境中极易复现。

修复建议：添加 const [submitting, setSubmitting] = useState(false) 状态，在 handleCreate 入口处判断 if (submitting) return，操作完成后恢复。

P0 #3 记账保存无防重复提交

位置：src/app/more/accounting/page.tsx，handleSaveRecord 函数（第 311-340 行）

风险：与首页问题相同。记账 BottomSheet 的保存按钮没有提交锁，反复点击"保存记录"将创建重复的 Transaction 记录，导致金额统计翻倍，严重影响记账数据准确性。

修复建议：增加 submitting 状态锁，并在 handleSaveRecord 中做入口判断。

P0 #4 17 个静默 catch 吞掉错误

概述：代码库中有大量 catch(err){} 块仅吞掉异常而不做任何日志或用户反馈，导致线上故障完全不可观测。以下是关键位置（共 17 处）：

P1 严重体验问题（7项）

P1 #1 记账明细列表不显示账户信息

位置：src/app/more/accounting/page.tsx，明细渲染区块（第 559-604 行）

问题：虽然代码中已有 accountMap 映射和 acc 变量渲染逻辑（第 583-586 行显示账户名称标签），但 t.accountId 字段在 addTransaction 调用中虽然传入了 selectedAccountId，在旧数据中 accountId 可能为空，导致多数记录不显示账户来源。且当前明细项更侧重显示分类颜色和备注，账户信息作为辅助信息字体过小（11px），在真实使用中容易被忽略。

修复建议：确保所有历史数据迁移填充 accountId 字段；将账户名称在明细列表中的展示权重提升，或在筛选器中增加"按账户筛选"能力。

P1 #2 缺少 prefers-reduced-motion 支持

位置：全局 framer-motion 动画配置

问题：LifeFlow 大量使用 framer-motion 进行页面转场和微交互动画（首页入场动画、BottomSheet 弹入弹出、列表项交错动画等），但未检测 prefers-reduced-motion 媒体查询。对前庭功能障碍或动画敏感的用户，这可能引发眩晕和不适。

修复建议：在 framer-motion 的 MotionConfig 或全局 CSS 中检测 @media (prefers-reduced-motion: reduce)，关闭所有非必要动画。framer-motion 提供了 useReducedMotion() hook 可直接使用。

P1 #3 各页面无独立 metadata title

位置：src/app/layout.tsx（第 15-24 行）

问题：根布局中仅定义了全局 title: "LifeFlow"，所有子页面（记账、日程、睡眠、饮水等）均未通过 generateMetadata() 或页面级 metadata 导出来设置独立标题。这导致用户在浏览器标签页中无法区分当前所在页面，也不利于 SEO 和 PWA 多页面体验。

修复建议：为每个页面添加 export const metadata: Metadata = { title: "记账 - LifeFlow" }；对于动态路由使用 generateMetadata()。

P1 #4 defaultLedgerId 为空导致记账报错（已修复但需标记）

位置：src/lib/db/accounting.db.ts，ensureDefaultLedger() 函数及 src/app/more/accounting/page.tsx 第 322-324 行

问题：在之前版本中，当用户首次打开记账页面时 ensureDefaultLedger() 返回的 ledgerId 可能为 undefined，导致 addTransaction 因 ledgerId 缺失而失败。本次会话中已通过自愈逻辑修复（在 handleSaveRecord 中调用 ensureDefaultLedger），但此修复依赖于运行时自愈，若 ensureDefaultLedger 自身异常（见 P0#4 中 accounting.db.ts:151 的静默 catch），问题仍会复现。

修复建议：加固 ensureDefaultLedger 的错误处理链路，增加重试和 fallback。

P1 #5 3 个占位页仍可访问

位置：

  • src/app/more/review/sleep/page.tsx — 睡眠复盘，显示"即将上线"
  • src/app/more/review/goals/page.tsx — 目标复盘，显示"即将上线"
  • src/app/more/review/finance/page.tsx — 记账复盘，显示"即将上线"
  • src/app/more/review/fitness/page.tsx — 健身复盘，显示"即将上线"
  • src/app/more/review/water/page.tsx — 饮水复盘，显示"即将上线"

问题：复盘总览页列出了 7 个模块入口，其中 5 个子模块详情页（睡眠/目标/记账/健身/饮水）均为占位页面，仅显示一条"即将上线"的文案和图标。用户点击后会感到困惑——既然功能未完成，为何在总览中展示入口？

修复建议：优先实现至少 2 个复盘子页面（如日程复盘 + 记账复盘）；对未实现页面的入口添加"即将推出"角标，或在总览页做条件渲染。

P1 #6 部分 useLiveQuery 无 loading 状态

位置：多个使用 dexie-react-hooks 的 useLiveQuery 的组件

问题：以记账页面为例，txs、categories、ledgers、accounts 等均通过 useLiveQuery(()=>..., [], [] as T[]) 获取，第三个参数为空数组 [] 作为兜底值。在 Dexie 数据库首次加载期间，这些值立即可用（空数组），页面直接渲染空列表而非 loading 骨架。这在数据库较大或 IndexedDB 初始化慢时会造成短暂的"无数据闪烁"，然后数据突然出现。

修复建议：利用 Dexie 的 useLiveQuery 返回的 undefined 作为 loading 判断（移除第三个默认值参数），或使用自定义的 useLiveQuery wrapper（已有 src/lib/hooks/useLiveQuery.ts 但未广泛使用）来统一管理 loading/error 状态。

P1 #7 SWProvider 中 SW 事件缺少 removeEventListener

位置：src/components/pwa/SWProvider.tsx，第 41-54 行

问题：在 registerSW 函数中，registration.addEventListener("updatefound", ...) 和 newWorker.addEventListener("statechange", ...) 注册了两个事件监听器，但 useEffect 的清理函数（第 82-85 行）仅清理了 online/offline 事件，未清理这两个 SW 事件。在 React Strict Mode（开发环境）下 useEffect 会执行两次，导致重复注册监听器；在组件卸载时也可能产生内存泄漏。

修复建议：将 registration 和 newWorker 的引用保存到 ref 中，在清理函数中调用 removeEventListener 移除。

P2 优化建议（12项）

#项目位置建议
1 useLiveQuery 未统一使用 src/lib/hooks/useLiveQuery.ts 该 hook 已实现 loading/error 状态管理，但各页面仍直接使用 dexie-react-hooks 的原生 useLiveQuery。建议全局替换为项目自定义版本。
2 记账页面未按分类筛选 src/app/more/accounting/page.tsx 明细列表仅支持关键词搜索，缺少按支出/收入类型和按分类的快速筛选器。
3 首页事项创建缺少分类选择 src/app/page.tsx，createForm 新建事项弹窗只支持标题、时间、备注和颜色，无法关联到具体目标或项目。
4 日程页面缺少拖拽排序 src/app/efficiency/schedule/page.tsx 时间轴事项为只读展示，无法直接拖拽调整时间或重新排序。
5 AgentProvider 过大（1200+ 行） src/components/agent/AgentProvider.tsx 建议拆分为多个子模块（消息管理、工具调用、流式处理、状态同步），降低单文件复杂度。
6 全局 CSS 变量未统一 src/app/globals.css 存在 var(--lifeflow-*) 和 var(--color-*) 两套命名空间混用，建议统一为单一前缀。
7 缺少 ErrorBoundary 细粒度覆盖 src/components/GlobalErrorBoundary.tsx 仅在根布局挂载了一个全局 ErrorBoundary，建议为记账、日程、代理人等关键模块添加独立 ErrorBoundary。
8 PWA manifest 图标单一 public/manifest.json 仅配置了单一图标，缺少 192px 和 512px 的 maskable 图标，影响 Android PWA 添加到主屏幕体验。
9 没有离线数据同步策略 全局 SW 实现了离线缓存，但 IndexedDB 数据与远程服务之间无同步机制。若未来引入云端备份，需提前设计冲突解决策略。
10 滚动性能优化空间 多个长列表页面 记账明细列表、日程时间轴等长列表未使用虚拟滚动（如 react-window 或 @tanstack/virtual），在数据量大时可能卡顿。
11 TypeScript strict 模式未开启 tsconfig.json 当前关闭了 strict 模式，存在不少隐式 any 和 undefined 风险，建议逐步开启。
12 缺少自动化测试 全局 整个项目无单元测试或 E2E 测试文件。建议至少为核心数据操作（记账 CRUD、事项增删改）添加单元测试。

Part 2: 产品逻辑与语言重构方案

产品语言定义

核心任务

LifeFlow 围绕三个核心任务构建：日程管理（日计划、时间轴、待办追踪）、习惯养成（饮水、睡眠、健身、吃药等日常行为的打卡与趋势追踪）、金融追踪（多账户收支记录、分类统计、月度/年度图表）。

核心对象

对象英文名定义所属空间
目标Goal用户设定的中长期目标，可拆解为多个事项目标、今日
事项Item日程中的最小执行单元，具有计划时间、完成状态今日、日程
项目Project分类容器，组织功能模块的入口（饮水、记账、睡眠等）项目管理
交易Transaction一笔收支记录，关联账本、账户、分类记账
药品Medicine用户服用的药品记录，关联剂量和提醒吃药

统一动词库

操作动词适用对象说明
创建新建目标、事项、交易、药品统一使用"新建"而非混用"创建"/"添加"/"新增"
完成完成事项勾选表示完成，已完成的不可再编辑
记录记录 / 记一笔交易、睡眠、饮水、健身行为数据的录入，尤其记账用"记一笔"
删除删除目标、事项、交易统一使用"删除"，避免混用"移除"
编辑编辑目标、事项、交易已创建条目的修改
打卡打卡习惯（饮水、吃药、健身）一次性完成动作，区别于"记录"（可能含数值）
回顾复盘目标、日程、记账、睡眠等统一使用"复盘"而非"回顾"/"查看报告"
校准校准日程调整计划与实际执行的偏差

消灭功能孤岛

当前孤岛问题

P2 孤岛 #1: 专注、倒数日、备忘录仅首页可到达

现状：/more/focus、/more/countdown、/more/notes 三个功能模块仅在首页右上角通过 <QuickBtn> 暴露入口（5 个小图标按钮），在项目管理页（/more/projects）中找不到对应入口。用户首次进入项目管理页时完全不知道这些功能的存在。

建议：在项目管理页的"工具"分类下添加这三个功能的入口卡片，确保全站导航图完整。

P2 孤岛 #2: 复盘子模块 5 个详情页全为占位

现状：/more/review/goals、/more/review/finance、/more/review/sleep、/more/review/water、/more/review/fitness 均为占位页面，仅展示"即将上线"文案，但复盘总览页将其作为正常入口展示。

建议：优先实现日程复盘和记账复盘两个子页面（数据已有），其余三个页面在总览页中添加"即将推出"标识。

P2 孤岛 #3: growth、learning、capture 三个无效路由

现状：

  • /growth 和 /learning 均客户端跳转到不存在的 /overview，导致 404
  • /capture 服务端 redirect("/") 跳回首页，功能等效于被阉割

建议：若这些功能短期内不实现，直接从路由表中删除并清理相关引用；若计划实现，创建占位页并标注"开发中"。

L1
Level 1: 立即修复（P0 + 导航骨架）
预计 4–6 小时
  • 修复 growth 和 learning 路由 404 — 改为 redirect("/") 或删除路由文件
  • 首页新建事项添加 submitting 防重复提交锁
  • 记账保存添加 submitting 防重复提交锁
  • 为 17 个静默 catch 块添加 console.error 和用户可见的错误提示（至少覆盖 DB 初始化和数据导出/导入路径）

  • 为 5 个主要页面（首页、日程、记账、助手、复盘）添加独立 metadata title
  • 删除或修复 /capture、/growth、/learning 三个死路由
L2
Level 2: 重要优化（P1 + 空间首页完善）
预计 8–12 小时
